# plugins/_template/

Scaffold for a new plugin's docs + landing subsite. Copy this directory, rename, substitute placeholders, customize, ship.

## Structure

```
site/
  index.html          # hand-crafted landing (bespoke per plugin)
  styles.css          # ~100 lines of layout + tokens
docs/
  zensical.toml       # Zensical config (Home / Quickstart / Architecture / Reference / Changelog)
  index.md            # docs home
  quickstart.md       # 5-min walkthrough
  architecture.md     # how it works
  reference/
    commands.md       # GENERATED from ../commands/*.md — do not hand-edit
  stylesheets/
    plugin.css        # per-plugin docs overrides (extend shared/tokens.css)
justfile              # docs-build / docs-stage / docs-serve / site-serve / prod-build
README.md             # this file (replace with plugin-specific README when instantiated)
```

## How to use

### Via `bin/new-plugin` (recommended)

From skills repo root:

```bash
bin/new-plugin orc            # creates plugins/orc/ pre-filled
```

The script copies this template, substitutes every placeholder, and stages the result. You still write the actual content — the template just gets you a running site scaffold to edit against.

### Manually

```bash
cp -r plugins/_template plugins/<name>
cd plugins/<name>
rg -l '{{PLUGIN' | xargs sed -i '' \
  -e 's/{{PLUGIN_NAME}}/<name>/g' \
  -e 's/{{PLUGIN_TAGLINE}}/<one-sentence tagline>/g' \
  -e 's/{{PLUGIN_DESCRIPTION}}/<longer description>/g'
```

## Placeholders

| Token | Where it appears | Example |
|---|---|---|
| `{{PLUGIN_NAME}}` | `<title>`, `<h1>`, install command, breadcrumb, zensical `site_name`, justfile banner, command tables. | `orc` |
| `{{PLUGIN_TAGLINE}}` | `<title>` subtitle, hero subhead. Short — 6–10 words. | `Multi-agent orchestration for Claude Code` |
| `{{PLUGIN_DESCRIPTION}}` | Meta description, landing lede paragraph. Longer — 1–2 sentences. | `Break complex work into parallel sub-tasks dispatched to Cursor, Codex, or Claude subagents. Wave-based execution, persistent state, clean handoff between sessions.` |

Also placeholder TEXT inside `docs/*.md` files (marked with `_Replace this..._`) — edit those freely.

## Customization points

**Keep generic (update at your peril):**
- `docs/reference/commands.md` — regenerated by `bin/gen-commands-doc`
- `docs/stylesheets/plugin.css` comment — keep the reference to `shared/tokens.css`
- `justfile` recipe names — `bin/build-plugin-site` expects `prod-build`

**Customize freely:**
- `site/index.html` content — after the header strip, it's all yours
- `site/styles.css` — brand palette, typography, hero layout
- `docs/stylesheets/plugin.css` — doc-site overrides (admonitions, code blocks, etc.)
- `docs/zensical.toml` nav structure — add sections that fit your plugin's shape
- Any `docs/*.md` body content

**Must-not-remove:**
- `<!-- HEADER_STRIP -->` marker in `site/index.html` (build-time injection point)
- `<link rel="stylesheet" href="../../shared/tokens.css">` (cohesion across subsites)
- The three generator-owned files: `docs/reference/commands.md`, `docs/changelog.md` (generated in CI), and anything containing the `generated by bin/` comment

## Visual identity

Folio went editorial (Libre Baskerville, parchment). Orc should go somewhere else — this template is intentionally neutral so you're forced to make a visual choice. Import `shared/tokens.css` for the baseline palette and type scale; override via `site/styles.css` for brand.

Don't try to make your plugin visually match folio or orc. The parent site ties them together via the header strip and token palette — that's enough cohesion. Beyond that, distinct per-plugin identity is the feature.
